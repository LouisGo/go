# File Protocol

The LouisGo protocol directory lives at the repository root:

```text
.louisgo/
├── MISSION.md
├── CAPABILITIES.md
├── STATE.md
├── CONTEXT.md
├── MEMORY.md
├── ROADMAP.md
├── HANDOFF.md
├── HANDOFF_DRAFT.md
├── QUICK_SAVE.md
├── BLOCKER.md
├── CONFIRM_REQ.md
├── RUNLOG.md
├── stats/
│   ├── events.jsonl
│   └── imports.json
├── test-results.json
├── memory/
├── sessions/
├── skills/
│   └── manifest.json
├── ADR/
│   └── draft/
└── scripts/
```

`louisgo init` creates only the minimal useful set: `MISSION.md`, `CAPABILITIES.md`, `STATE.md`, and `.gitignore`. Other files and directories are created by the command or real content that needs them, so fresh projects do not inject template prompt into context.

`louisgo verify` runs from the global CLI by default and writes `.louisgo/test-results.json`. If the project has no real verification command and no explicitly maintained `.louisgo/scripts/verify.*`, the result is `skipped`. Project-local verification scripts are compatibility or advanced opt-in files and are no longer generated by default.

`HANDOFF_DRAFT.md` and `QUICK_SAVE.md` are legacy compatibility files, not primary recovery sources.

`louisgo clear` shows risk and cleanup targets, then requires a direction-key selection prompt before deleting this Git project's `.louisgo/`. It also removes the LouisGo-managed Codex block from project agent instruction files. Use `louisgo clear --dry-run` to preview targets. The command does not delete product source code, global Codex config, or global skills.

## Protocol Role

`.louisgo/` is a Git-synced prompt cache, not a full knowledge base.

Protocol files should be:

- Human-readable: Markdown bodies can be reviewed directly.
- Machine-readable: front matter provides stable schema.
- Layered and trim-friendly: base rules are short and stable; active state is short and fresh; details are read on demand.
- Syncable: files can move through Git across machines, worktrees, and agents by default.
- Verifiable: source code facts and verification results override memory narratives.

## Format

Markdown files use YAML front matter by default:

```markdown
---
schema: louisgo-state-v1
mode: assist
current_task: T001
handoff: .louisgo/HANDOFF.md
verification: stale
git_head: abc123
diff_hash: def456
updated_at: "2026-05-02T12:00:00.000Z"
---

# State
```

Rules:

- Front matter gives the CLI and AI a stable read surface.
- Body text is for humans and AI to read and edit.
- JSON is only for machine verification results such as `test-results.json`.
- AI may update `STATE.md`, `MEMORY.md`, `sessions/*.md`, `HANDOFF.md`, `CONFIRM_REQ.md`, and ADR drafts, but must not fake a passing verification result.

## Core Files

### `MISSION.md`

Project goals, technical constraints, forbidden actions, confirmation rules, and default mode. AI must respect it during every recovery.

### `CAPABILITIES.md`

Current repository verification entry and behavior contract. By default it records global `louisgo verify` behavior and platform integration state.

### `STATE.md`

Current working state. This is L4 active cache: short, small, and fresh enough for each new session to read automatically.

Suggested structure:

```markdown
# State

## Now

Current task, workspace state, and first priority.

## Next

The first concrete next action.

## Recovery Order

1. HANDOFF.md
2. STATE.md
3. MEMORY.md
4. memory/*.md
5. sessions/*.md
```

### `MEMORY.md`

Rolling memory index. This is L2 stable index and should not become a long chat log. It records durable conventions, topic files, and recent session pointers.

Suggested structure:

```markdown
# Memory

## Stable Notes

- Durable product and engineering conventions.

## Topic Files

- memory/architecture.md
- memory/testing.md

## Recent Sessions

- sessions/2026-05-02-xxxx.md
```

### `CONTEXT.md`

Optional project domain glossary. If present, AI uses the definitions during context recovery to avoid synonym ambiguity.

This file has no front matter. AI or users create it on demand; `louisgo init` does not generate it.

Suggested structure:

```markdown
# {Project Name} Domain Glossary

## Language

- **Term**: definition _Avoid_: alias1, alias2

## Relationships

- term A relates to term B because...

## Flagged Ambiguities

- term X was used to mean both Y and Z - resolved: ...
```

### `skills/`

Behavior-guidance skill directory. `louisgo init` does not write behavior skills by default. Users inspect presets with `louisgo skill list`, then enable only what is useful with `louisgo skill enable grill` or `louisgo skill enable caveman`.

Preset skills:

- **grill.md**: pressure-test each branch of a user's plan until the tradeoffs are jointly understood. Use when the user says "grill me" or wants design validation.
- **caveman.md**: ultra-compressed communication that removes filler while preserving technical accuracy. Use when the user says "caveman mode" or wants fewer tokens.

Rules:

- Skill content is English because AI reads it.
- Enabling or disabling a LouisGo-managed skill refreshes `.louisgo/skills/manifest.json`.
- The manifest is the discovery index: Codex reads metadata first, then lazy-loads only the matched skill file when the user invokes a skill by name, alias, or trigger phrase.
- If the manifest is absent, Codex falls back to scanning `.louisgo/skills/*.md` and `.louisgo/skills/*/SKILL.md` front matter.
- The manifest has a reserved Claude slot so a future `CLAUDE.md` or plugin manifest adapter can use the same skill metadata.
- Before enabling a skill, LouisGo checks `.codex/skills/` and `.louisgo/skills/` for same-name skills. If it finds a conflict, it reports it and does not overwrite project content.
- `louisgo skill disable <name>` only removes LouisGo-managed preset files. It does not delete user-maintained files.
- Skill bodies are not auto-injected into context. `louisgo context` includes at most the local skill index.

### `HANDOFF.md`

Formal handoff snapshot. This is L3 formal recovery cache and takes priority over ordinary memory when present.

It must include:

- Current task and mode.
- Git HEAD and diff hash.
- Verification status.
- Current workspace summary.
- Blockers, confirmation requests, and ADR drafts.
- Next recovery advice.

### `CONFIRM_REQ.md`

Open confirmation request. If present, AI must surface it first and wait for the user's choice. It must not be buried in ordinary memory.

### `RUNLOG.md`

Local diagnostic log. LouisGo commands append short events containing command name, result, recovery source, verification state, confirmation request signal, ADR draft count, and workspace summary.

Rules:

- Do not record user prompt text, full chat content, source code content, or secrets.
- Ignore it by default through `.louisgo/.gitignore` to avoid Git noise.
- For diagnostics, run `louisgo log --tail 30` or send `.louisgo/RUNLOG.md`.
- Keep only recent events to avoid long-term token growth.

### `stats/`

Local token and context observation directory, ignored by `.louisgo/.gitignore` by default.

Files:

- `events.jsonl`: LouisGo context estimation events and explicitly imported Codex usage events.
- `imports.json`: Codex JSONL file fingerprints to avoid duplicate imports.

Rules:

- `louisgo context` writes only section names, source paths, token counts, and cache-friendly metrics.
- `louisgo stats import codex` extracts only numeric fields such as `input_tokens`, `cached_input_tokens`, `output_tokens`, `reasoning_output_tokens`, and `total_tokens`.
- Do not store user prompts, full chats, model response bodies, source code, or secrets.
- Stats categories are `actual`, `estimated`, and `simulated`: real Codex usage, local tokenizer estimates, and baseline-based simulated savings.

### `test-results.json`

Verification fact source. Minimum fields:

- schema
- command
- status
- exit_code
- started_at / finished_at
- git_head
- diff_hash
- summary

`status` values:

```text
passed | failed | error | skipped
```

The CLI additionally derives `missing` and `stale` from Git HEAD and diff hash.

The diff hash ignores generated LouisGo recovery and diagnostic files: `test-results.json`, `RUNLOG.md`, `HANDOFF.md`, `HANDOFF_DRAFT.md`, `QUICK_SAVE.md`, `STATE.md`, `CONFIRM_REQ.md`, `stats/`, and `sessions/`. This prevents `$finish`, log append, and local stats events from making a fresh verification immediately stale. Source code, verification scripts, and non-ignored protocol files still affect freshness.

## Cache Layers

| Layer | Files | Writer | Change frequency | Git sync |
| --- | --- | --- | --- | --- |
| L0 Platform entry | Project agent instruction files and future platform adapters | `louisgo init` / user | Low | Yes |
| L1 Project contract | `MISSION.md`, `CAPABILITIES.md` | Mostly user, AI-assisted | Low | Yes |
| L2 Stable index | `MEMORY.md`, `CONTEXT.md`, `skills/manifest.json`, `skills/*.md`, `memory/*.md` | AI / user | Low | Yes |
| L3 Formal recovery | `HANDOFF.md` | `$finish` / AI | Phase-based | Yes |
| L4 Active state | `STATE.md`, `CONFIRM_REQ.md`, `sessions/*.md` | AI | Frequent | Yes, with project-specific session policy if needed |
| L4b Local observation | `stats/*.jsonl`, `stats/imports.json` | CLI | Frequent | No by default |
| L5 Task skills | Codex skills, future platform skill/rule files, `.louisgo/skills/*` | `louisgo skill` / user | Workflow-dependent | Platform-specific or project-specific |

Prompt assembly should keep L0-L2 as a stable prefix when possible and place L3-L4 later. Semantic recovery still uses confirmation request, formal handoff, and active state priority.

## State Priority

Semantic recovery uses this order:

1. `CONFIRM_REQ.md`: handle confirmation first when present.
2. `HANDOFF.md`: formal handoff has priority.
3. `STATE.md`: current state.
4. `MEMORY.md`: rolling memory index.
5. `memory/*.md` and `sessions/*.md`: read only as needed for the task.
6. Git, source code, and verification results: confirm facts.

If file content conflicts with repository facts, trust source code, Git, and verification results, then update the relevant protocol file.

## Subagent Context Capsule

When the main agent delegates to a subagent, it should not pass the full memory library. LouisGo recommends `louisgo context --capsule --goal "<task>"` to generate a short capsule:

```markdown
---
schema: louisgo-context-capsule-v1
task_id: T001
mode: assist
source_handoff: .louisgo/HANDOFF.md
created_at: "2026-05-03T12:00:00.000Z"
---

# Context Capsule

## Goal

## Scope

## Relevant Files

## Constraints

## Verification

## Expected Output
```

The capsule can be a temporary prompt or can be written to `.louisgo/sessions/` when auditability is needed. Its purpose is to give the subagent clean context, not to create long-term memory.
