# MVP 产品需求

## 目标

构建一个轻量级 AI 编程 Harness，使它可以被加入到任意软件项目中，并且不绑定具体语言、框架、构建工具或测试工具。

第一版优先服务 Codex CLI。长期方向是沉淀为一套仓库内协议，让其他 AI 编程工具也可以遵循同一套工作范式。

## 问题背景

AI 编程会话的可靠性经常下降，核心原因是关键状态过于隐式：

- AI 可能声称任务已经完成，但缺少可验证证据。
- 上下文散落在聊天记录中，而不是沉淀在项目文件里。
- 会话交接依赖叙事总结，容易和真实仓库状态漂移。
- 用户很难一眼看出 AI 当前是在自动推进、协助执行，还是只读观察。
- 暂停和恢复工作需要重新解释上下文，成本过高。

这个 Harness 的目标是把 AI 工作流状态显式落到文件里，并通过确定性校验降低状态漂移。

## 产品目标

- 提供一套足够小的仓库内约定，可跨技术栈复用。
- 让重要 AI 工作流状态在文件中可见。
- 区分人类可读的项目意图和机器可校验的事实。
- 让用户清楚控制 AI 的自主程度。
- 支持轻量暂停和恢复，不要求每次都做完整交接。
- 让正式交接先生成草稿，经用户确认后再成为权威记忆。

## 非目标

- MVP 不构建守护进程或后台服务。
- MVP 不要求项目使用特定应用框架、包管理器或测试框架。
- MVP 不替代 Git、CI 或项目已有工程工具。
- MVP 不实现复杂多 Agent 编排。
- MVP 不把 AI 自述当作正确性证据。
- MVP 不要求所有项目使用同一条测试命令。

## 语言约定

- 项目文档使用简体中文。
- 后续代码注释默认使用简体中文。
- 命令名、文件名、字段名、JSON schema 值等协议标识可以使用英文，以保证跨工具稳定性。
- 面向用户的提示语优先使用简体中文。

## MVP 范围

MVP 支持一个 AI 编程会话在一个仓库内工作。

必须支持的能力：

- 启动会话时读取项目约束和上次交接状态。
- 为短时中断保存轻量进度。
- 从轻量进度或正式交接状态恢复。
- 结束会话时生成可供用户审阅的交接草稿。
- 运行项目自定义的验证脚本。
- 将验证结果记录为机器可检查的结果文件。
- 维护一个简单的任务路线图。
- 在对话中显示当前 AI 自主模式。
- 在出现歧义或关键选择时，生成可见的确认请求。
- 为架构决策保留可审阅的 ADR 草稿和正式记录。

暂缓到后续版本的能力：

- 自动委派任务给更快或更低成本的 AI 模型。
- 自动创建和合并 Git worktree。
- 多 Agent 调度。
- Pull Request 自动化。
- Codex CLI 之外的工具深度集成。

## 指令命名

Harness 指令统一使用 `$` 前缀。

第一版指令集：

| 指令 | 含义 | MVP 是否实现 |
| --- | --- | --- |
| `$start` | 开始工作并加载项目上下文。 | 是 |
| `$pause` | 为短时中断保存轻量进度。 | 是 |
| `$resume` | 从正式交接状态恢复。 | 是 |
| `$finish` | 结束会话并生成交接草稿。 | 是 |
| `$delegate <任务>` | 委派子任务给其他 AI 或隔离工作区。 | 后续 |
| `$merge <标识>` | 合并被委派任务的结果。 | 后续 |
| `$review <范围>` | 启动独立审查。 | 后续 |
| `$decide` | 主动触发决策确认。 | 是，MVP 支持文件协议；CLI 命令后续实现 |

命名原则：

- 指令应该像用户动作，而不是实现细节。
- `$` 比 `@` 更适合作为显式命令前缀，也更贴近 Codex CLI 的使用习惯。
- MVP 只实现可靠单会话流程所需的基础指令和文件协议。

## 自主模式

Harness 使用三个模式：

| 模式 | 含义 | 允许行为 |
| --- | --- | --- |
| `auto` | AI 主动推进任务。 | 可以编辑文件、触发验证、更新路线图，并在小步实现中继续推进；但不能仅凭自述判定任务完成。 |
| `assist` | AI 协助执行，但关键点先确认。 | 可以编辑文件，但在架构方向、大范围重构、风险变更或不可逆工作流步骤前必须询问用户。 |
| `manual` | AI 只读观察和建议。 | 除非用户明确要求，否则不能修改文件或运行会改变状态的命令。 |

说明：

- `assist` 表示协助执行模式。
- `auto` 和 `manual` 保持不变。
- 面向用户的状态提示必须包含当前模式，例如：`[assist] 需要先确认实现方向。`

## 质量门禁

MVP 必须明确区分“触发验证”和“判定质量”：

- 生成代码的 AI 可以触发 `louisgo verify` 或项目验证脚本。
- AI 不能把自己的叙事总结作为完成依据。
- 任务完成只能来自可验证事实、独立审查或用户确认。
- 架构决策必须先进入 ADR 草稿，再由用户确认成为正式 ADR。

## 必需文件

MVP 约定所有 Harness 文件放在 `.louisgo/` 目录下：

| 文件 | 作用 | MVP 要求 |
| --- | --- | --- |
| `.louisgo/MISSION.md` | 人类编写的项目规则和约束。 | 必需 |
| `.louisgo/ROADMAP.md` | 简单任务清单。 | 必需 |
| `.louisgo/HANDOFF.md` | 用户确认后的权威恢复点。 | 首次运行可不存在 |
| `.louisgo/HANDOFF_DRAFT.md` | AI 生成的正式交接草稿。 | 由 `$finish` 生成 |
| `.louisgo/QUICK_SAVE.md` | 轻量暂停状态。 | 由 `$pause` 生成 |
| `.louisgo/BLOCKER.md` | 可复现阻塞和验证失败记录。 | 必需 |
| `.louisgo/CONFIRM_REQ.md` | 临时决策请求和歧义确认卡片。 | 按需生成，存在时必须处理 |
| `.louisgo/CAPABILITIES.md` | 可用脚本和行为约定。 | 必需 |
| `.louisgo/test-results.json` | 机器可检查的验证结果。 | 由验证脚本生成 |
| `.louisgo/ADR/draft/` | 架构决策草稿目录。 | 必需 |
| `.louisgo/ADR/*.md` | 用户确认后的正式架构决策。 | 按需生成 |
| `.louisgo/scripts/verify.sh` | macOS / Linux 项目自定义验证入口。 | 必需 |
| `.louisgo/scripts/verify.ps1` | Windows 项目自定义验证入口。 | 由脚手架生成 |

委派相关文件和脚本等单会话流程稳定后再引入。

## 验证要求

验证结果必须被视为事实来源，而不是叙事总结。

MVP 的验证结果至少应该包含足够信息，用于判断结果是否过期：

- Schema 版本。
- 实际运行的命令。
- 退出码。
- 开始时间和结束时间。
- Git HEAD。
- 工作区 diff hash 或等价的 dirty-state 指纹。
- 人类可读摘要。

`$finish` 不能静默依赖过期验证结果。如果验证结果和当前仓库状态不匹配，交接草稿必须明确标记“验证结果已过期”，或者先重新运行验证。

## 决策确认要求

当出现以下情况时，AI 必须生成 `.louisgo/CONFIRM_REQ.md`，并等待用户选择：

- 指令存在两种或以上明显不同的架构方向。
- 指令与 `MISSION.md` 中的核心约束或禁止事项冲突。
- 修改范围不明确，且可能影响多个模块或公共接口。
- AI 需要用户在风险、范围、兼容性之间做取舍。

确认请求被解决后，AI 应删除 `.louisgo/CONFIRM_REQ.md`。如果用户在确认前执行 `$finish`，确认请求必须转存到 `HANDOFF_DRAFT.md` 的遗留问题中。

## ADR 要求

当任务涉及以下内容时，AI 必须生成 ADR 草稿：

- 公开 API 或协议变更。
- 数据模型或持久化格式变更。
- 新技术栈、关键依赖或架构分层变更。
- 影响多个模块边界的设计决策。

ADR 草稿放在 `.louisgo/ADR/draft/`。用户确认后，才能成为 `.louisgo/ADR/NNN-title.md` 形式的正式 ADR。

## 会话流程

### `$start`

预期行为：

- 读取 `.louisgo/CAPABILITIES.md`。
- 读取 `.louisgo/MISSION.md`。
- 如果 `.louisgo/CONFIRM_REQ.md` 存在，则优先提示存在未解决确认请求。
- 如果 `.louisgo/QUICK_SAVE.md` 存在且比 `.louisgo/HANDOFF.md` 更新，则优先读取轻量保存状态。
- 否则读取 `.louisgo/HANDOFF.md`，如果该文件存在。
- 如果没有正式交接状态，则从 `.louisgo/ROADMAP.md` 的第一个未完成任务开始。
- 显示当前自主模式。

### `$pause`

预期行为：

- 写入 `.louisgo/QUICK_SAVE.md`。
- 记录当前模式、当前任务、下一步和相关仓库状态。
- 不清空阻塞记录。
- 不生成正式交接。
- 除非完成状态已经通过验证，否则不勾选路线图任务。

### `$resume`

预期行为：

- 从 `.louisgo/HANDOFF.md` 重新加载。
- 忽略 `.louisgo/QUICK_SAVE.md`。
- 显示当前模式和下一项任务。

### `$finish`

预期行为：

- 生成 `.louisgo/HANDOFF_DRAFT.md`。
- 写入当前模式、变更摘要、验证状态、阻塞项、未解决决策和下一步具体行动。
- 如果 `.louisgo/CONFIRM_REQ.md` 存在且未解决，将内容转存到交接草稿的遗留问题中。
- 只有在交接草稿已经捕获相关状态后，才清理 `.louisgo/QUICK_SAVE.md`。
- 不直接写入 `.louisgo/HANDOFF.md`；必须由用户确认草稿后再成为权威交接状态。

### `$decide`

预期行为：

- 生成或更新 `.louisgo/CONFIRM_REQ.md`。
- 列出清晰选项、风险和建议默认选项。
- 等待用户选择后再继续执行。
- MVP 只要求文件协议和 AI 行为生效，不要求实现独立 CLI 命令。

## 已确认决策

- `.louisgo/ROADMAP.md` 必须使用稳定任务 ID。
- `.louisgo/HANDOFF.md` 不通过手动复制生成；优先由 AI 基于模板生成，必要时提供脚本提升草稿。
- `.louisgo/test-results.json` 的最小 schema 由协议文档定义。
- `.louisgo/CONFIRM_REQ.md` 和 `.louisgo/ADR/draft/` 进入 MVP 协议，但独立 CLI 命令可以后续实现。
- `$start` 遇到缺失的 `.louisgo/` 文件时，先报告缺失，再提醒用户将自动创建。
- `manual` 模式允许低风险只读检查，遵循业内常见最小打扰原则；会改变文件、Git 状态、网络状态或外部系统状态的命令必须先获得用户明确指令。
- MVP 需要提供命令行脚手架，避免用户手动复制 `.louisgo/` 模板。
- MVP 从第一版开始保证 Windows 基础可用，但主要针对 macOS 体验调优。
